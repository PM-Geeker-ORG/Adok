# 产品的自我修养
作为产品应该经常反省自己的产品修养，不断学习和实践，成为真正的产品人。这里将会分享关于产品的笔记内容，可能无法做到非常体系也无法做到非常深刻，但会涉及到产品的多个方面，**希望大家在不足中可以遇见美好**。

> **关于New Bing（GPT-4）的对话**

Q：世界为什么需要产品经理？

A：产品经理是一个负责设计和管理产品的角色，他们需要了解用户的需求，分析市场的趋势，制定产品的目标和策略，协调开发和运营团队，不断优化和迭代产品，为用户提供更好的体验和价值。**```世界为什么需要产品经理呢？```** 因为产品经理可以**帮助创造更多有用、有趣、有影响力的产品，解决用户的痛点，满足用户的期望，推动社会的进步和创新**。

## 🔅 产品人的思维

> **```产品思维是产品最基础的东西，它具有抽象性，当一个产品人沉淀了自己的产品认知体系后，它又会变得现实性。但是由于变化才是永远不变的东西，所以产品思维也是一个动态的思维，产品人需要不断完善和创新，让产品越来越好```**

## 🔅 产品人的日常

> **```产品的日常工作属于多领域。如果在大团队，由于分工，可能会限定于几个固定的工作流；如果在小团队，可能就不再细分，让产品多方位执行。但是由于产品的定位是团队的核心，所以不断怎样，产品人的日常都需要在多领域中思考着工作。```**

| 工作名称 | 工作内容 |
| :---: | :--- |
| -- |  |
| **需求收集** | 收集来自外部和内部的产品需求情况并汇集到需求清单 |
| **需求评审** | 向团队相关成员进行需求文档的相关内容阐述和说明以便目标一致 |
| **需求分析** | 对收集后的需求进行合理思考分析和取舍并设置需求优先级 |
| **数据分析** | 运营中所收集到的数据进行适当的统计分析以便优化或创新 |
| **竞品分析** | 对目前主要竞争或龙头产品进行优势和劣势分析以便学习和创新 |
| **行业分析** | 对产品所在的行业进行调研并从中了解到行业的发展趋势和方向 |
| **商业分析** | 从经济效益上分析自身产品的商业模型和商业盈利以便改进 |
| -- |  |
| **版本管理** | 制定合理的产品发展路线并以版本形式进行管理和迭代更新 |
| **业务规划** | 随着市场或商业的变化需要不断规划自身产品的业务情况以作适应 |
| **沟通协调** | 对团队中不同部门的人员进行沟通和协作以致为达到工作效率 |
| **项目管理** | 规划项目全局情况并进行项目资源最优化的把控和按时交付 |
| **项目复盘** | 进行项目途中或结束的事实陈述并分析总结获得改进的收获 |
| -- |  |
| **文档撰写** | 撰写各种演示文档或需求文档以作为留档或供团队成员阅读 |
| **市场调研** | 收集和分析市场情况以便预测市场的方向为后续产品发展做准备 |
| **用户调研** | 对用户画像、喜好、习惯和反馈情况进行调研以便更亲近用户的心智 |
| **用户体验** | 能够站在用户的角度去体验用户的需求、情感、行为和反馈情况 |
| **产品设计** | 从真实的需求出发为产品的业务或流程设计出合理的使用价值 |
| **产品规划** | 基于产品现状和目标对产品制定良好的未来行动路径和计划方案 |
| **产品运营** | 对上线产品的运营维护以使产品得到良好运作和达到既定目标 |
| **产品优化** | 调整产品存在的问题或不足并持续执行优化设计和产品迭代 |
| **产品创新** | 为适应产品发展对原产品的功能进行创新或重新开发新款产品 |
| **产品验收** | 对待上线的产品进行验收测试确保产品流程达到闭环并可正常上线 |

**✅ 简单的产品工作流24步：**

1️⃣ ```Start:需求来源``` => **需求池记录** => **优先级评定** => **版本迭代需求确定** => **立项（可延后至评审之后）**=>

2️⃣ **原型设计** => **评审第一次** => **评审第二次** => **设计稿** =>

3️⃣ **设计稿评审** => **开发人员分工** => **开发排期** => **需求文档（可提前到评审之前）**=>

4️⃣ **数据库及开发接口文档评审** => **项目开发** => **产品开发沟通协助** => **测试用例** =>

5️⃣ **测试用例评审** => **测试环境搭建** => **提测** => **修复Bug** =>

6️⃣ **产品上线前验收** => **正式环境搭建** => **产品上线（可增加灰度上线）**=> **正式环境验收** => ```End:需求来源```

| **跨界模式** | 打破了原有的行业特征，高效率地整合几个行业来提升整体系统效率和优势 |
| -- | |
| **金融理财** | 将钱借给金融机构，金融机构用这笔钱去赚取更多利润，并以利息的方式返还利润 |

## 🔅 竞品分析

**✅ 三个基础问题**

1. ❗什么是竞品分析？

> **系统地对竞争对手的优势和劣势进行分析评价**

2. ❗竞品分析目的？

> **帮助自身产品实现市场功能；产品设计和可用性提供参考；提高自身产品的差异化**

3. ❗为什么做竞品分析？

> **为企业制定产品战略规划；了解竞争对手的产品和市场动态；掌握竞争对手的资本背景和产品运营策略；快速自我调整并提供市场占有率；形成有效的系统化思想和客观准确方向等**



**✅ 竞品分析怎么做**

1. **三个方面**

描述性（说明我们和竞争对手有哪些差距）

定性分析（说明我们和竞品的差距在哪里） 

定量分析（说明我们和竞争对手的差距有多大||功能价值=功能得分*级别得分*权重值）

2. **分析维度**

战略定位BRD；商业定位MRD；用户群体MRD；产品功能PRD；产品界面（交互方式和视觉）PRD

3. **竞品选择**

直接竞争者；间接竞争者；潜在竞争者

4. **分析方法**

SWOT分析（优势、劣势、机会、威胁）四大板块定量评分表格；

竞争分析模型SAPPEALS：价格 可获得性 包装 性能 易用性 保证 生命周期 社会接受度

5. **获取渠道**

从内部市场、运营部门、管理层等收集信息；

行业媒体平台新闻及论坛；

建立持续的产品市场信息小组；

调查核心用户、活跃用户、普通用户不同需求；

竞争对手网站、交流互动平台、促销活动；

季表和年表财报；各大人才招聘网站信息和招聘更新；

通过谷歌搜索了解国外同行业信息和盈利模式；

6. **获取内容**

公司技术、产品、运营团队规模及核心目标和行业品牌影响力；

实际季度年度盈利和公司主盈利的产品线；

用户群体覆盖面和市场占有率；

产品功能稳定性、易用性、用户交互和技术框架细分及对比；

产品平台及官方的排名和关键字及外链数量

## 🔅 数据分析

1. 思路：基于用户路径（操作、流失、停留时间），基于产品节点（转化率，占比）
2. 分析类型：定性（对事物性质的归纳、是什么的假设），定量（对事物数量的统计、有多少的验证）
3. 步骤：定义（针对问题是什么、分析最终达到什么目的、对产品有何实际意义、确定分析范围、规划分析进度和质量），测量（收集获取数据、数据预处理），分析（数据统计描述、针对问题的归纳和总结），改进（找到最优的解决方法、问题解决或负面影响降低），控制（坚持监控和反馈、跟踪迭代）
4. 数据来源：自己产品（产品运营数据、用户反馈和调查数据），竞争产品（网站流量、公司财报），行业（行业分析报告（艾瑞 / 百度统计）、热点大数据披露）
5. 关键指标：转化率、任务完成率、当前使用用户数、新用户数、流失用户数、回访用户数、活跃用户数
6. 分析方法：时间序列趋势分析，AHP层次分析法（对定性问题进行定量分析的多准则决策方法、多目标决策和多个影响指标评价各方面方案优劣程度），用户忠诚度分析（用户使用频率、最近使用时间、平均使用时长、平均使用页面数），数据透视表（excel表格）
7. AHP层次分析法：横纵相同字段作为矩阵A 求纵向的每个占比 再横向合计行值W 最后以对比个数为参数乘以横线合计值 … 再获得CR=CI/RI CR<0.1则符合一致性

## 🔅 产品运营

**最高境界让产品自己运营自己**

1. 运营：产品，用户，渠道
2. 目标：收入，用户量，用户活跃度
3. 运营思路：成本和产出ROI，聚焦和细分（目标用户群），效率和量级（用户总量，AURP单用户收益），铺垫和节奏
4. 核心任务：吸引用户（开源），留住客户（节流）
5. 运营分类：用户运营，内容运营，活动运营，产品营销推广

用户运营：以产品用户为核心制定运营方案甚至运营机制，任务（掌握用户情况、了解用户分级、熟悉用户习惯、懂得用户来龙去脉），核心（开源-拉新、节流-挽留、维持-促活、刺激-付费），改善用户计产品情况（建立完善帮助系统、及时解决用户反馈问题、主动邀请用户体验和测试、对产品进行迭代优化）

内容运营：通过内容提高产品价值和促进用户粘性，任务（创作内容或者运营文章、编辑材料和组织专题、EDM电子邮件营销和新媒体都是内容运营、运营分析找到用户兴趣点并优化内容），核心（持续制作有价值内容、保证用户获取内容信息、根据KPI降低用户获取内容的成本、协助公司或产品获利）

活动运营：通过独立或者联合的活动拉动某一或一些指标的短期提升，任务（策划活动并编写文案、了解指标的成本和把控风险、结构性思维和数据驱动），引导用户创造内容，提高用户活跃度，对外合作提高产品影响

产品运营推广：平台推广，合作推广，物料推广，技术营销，广告，网络公关

6. 策划活动：规则（流程简单少思考、文档清晰无歧义），思路（特定时间、产品或商品本身、时会热点、自爆），活动策划书（版本控制，主题、对象、时间、活动描述、规则、投放渠道、风险控制防作弊、检测指标、成本预估、效果评估、FAQ常见问题解答）

## 🔅 产品思维和技术思维

1. 产品思维：从用户价值出发，在满足商业策略和业务目标的同时寻求产品路径满足用户需求（用户价值、使用场景、商业价值、业务闭环）
2. 技术思维：从功能和工程实现出发，在满足产品需求的同时寻求可复用技术架构和低开发成本（实现方式、技术架构、技术价值、开发成本）
3. 优秀产品：扎实的技术基础，良好的用户体验（产品思维先行，技术思维是确保产品能成为现实的有利前提和必要工具）
4. 技术是工具，是实现产品构思的一种方式，不必把技术当成产品的拦路虎（技术思维是一种严谨的逻辑思维，强调历遍和穷举）
5. 前端（动画、跳转实现设计），前后端（数据接口API设计），后端（数据库设计）
6. 程序 = 数据结构 + 算法  结构结构未数据类型 算法为规则
7. 懂技术不如懂产品，技术把握理性，产品把握人性
8. 产品上线时需要注意的问题：新老版本产品的交叉测试，确保功能兼容性和数据兼用型，产品后台配置文件是否已经更新到最新状态，数据接口是否能实现对新老版本的兼容，上线先后顺序的控制策略保证服务器先上线
9. 服务端和客户端沟通构成完整的产品技术体系
10. 服务端是产品实现数据交换和信息传递的中间调度者
11. 客户端和服务端之间通过JSON/XML进行数据接口的交流
12. 客户端和服务端之间基于请求响应模型进行交互
13. 云服务器作为中心化处理和数据中心为产品提供技术和支撑
14. Android和IOS基本控件：文本展示框 A-TextView I-UILabel，按钮 A-Button I-UIBotton，输入输入框 A-EditText I-UTextField，图片展示框 A-ImageView I-UIImageView，列表展示容器 A-ListView I-UITableView，表格展示容器 A-Gridview I-UICollectionView
15. Android应用特点：安装文件扩展名.apk，手机尺寸多样化适配工作量大，系统开源可定制化系统，应用市场碎片化严重渠道多样，手机硬件跨度大应用支持情况多样
16. IOS应用特点：安装文件扩展名.ipa，手机尺寸相对单一适配工作适中，系统闭源不可定制化系统，官方指定应用市场渠道单一，手机硬件差异小应用支持情况统一
17. 界面布局：各种UI控件按照一定的布局规则组合在一起，构成一个独立的产品界面
18. 软件测试包括：数据和数据库完整性测试：数据与数据库完整测试是指测试关系型数据库完整性原则以及数据合理性测试。

黑盒测试：黑盒测试也称功能测试

白盒测试：白盒测试是基于代码的测试

UI测试：UI测试指测试用户界面的风格是否满足客户要求

功能测试：功能测试指测试软件各个功能模块是否正确和逻辑是否正确

性能测试：性能测试主要测试软件测试的性能，包括负载测试，强度测试，数据库容量测试，基准测试以及基准测试

安全性和访问控制测试：应用程序级别的安全性，包括对数据或业务功能的访问；系统级别的安全性，包括对系统的登录或远程访问

故障转移和恢复测试：故障转移和恢复测试指当主机软硬件发生灾难时候，备份机器是否能够正常启动，系统是否可以正常运行

配置测试：配置测试核实测试对象在不同的软件和硬件配置中的运行情况亦称兼容性测试
